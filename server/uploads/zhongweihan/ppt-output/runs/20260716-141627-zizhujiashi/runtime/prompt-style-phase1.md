## 1. 作用定位

- 给 Step 3.5 风格产出，以及 Step 4 planning / HTML 提供统一的风格字段合同。
- 约束 `style.json` 的写法，让风格信息能够被 packet 和 prompt 稳定消费。
- 只定义设计目标、变奏边界和违约条件，不定义某一页应该长成什么样。

## 2. `style.json` 字段合同

| 字段 | 必填 | 约束 |
|------|------|------|
| `style_id` | 是 | 机器可读 ID；允许使用预置风格 ID 或项目自定义 ID，但必须稳定、可复用。 |
| `style_name` | 是 | 面向人类阅读的风格名称；用于识别，不用于驱动具体页面长相。 |
| `mood_keywords` | 是 | 3-5 个关键词；描述气质和阅读感受，不写色号，不写具体场景镜头。 |
| `design_soul` | 是 | 一句话说明整套 deck 想建立的设计目标和阅读氛围；禁止写成某一页的成品描述。 |
| `variation_strategy` | 是 | 描述跨页变化原则；必须说明哪些维度可变、哪些维度必须保持一致。 |
| `decoration_dna.signature_move` | 是 | 1 个跨页可重复识别的标志性动作；必须可抽象、可复用。 |
| `decoration_dna.forbidden` | 是 | 2-5 个会破坏一致性的禁用方向；写失败边界，不写风格恐吓。 |
| `decoration_dna.recommended_combos` | 是 | 2-4 个装饰组合语法；写组合关系，不写页面成品。 |
| `css_variables` | 是 | 全局变量真源；颜色、卡片、文字、圆角等关键变量必须完整。 |
| `font_family` | 是 | 字体栈；必须兼顾可用性和跨页统一。 |
| `css_snippets` | 否 | 仅用于固化必须统一的非颜色样式锚点；不能塞进整页骨架。 |

---

## 3. 字段写法规则

### `mood_keywords`

- 必须描述整体感受和读者接收状态。
- 不写具体颜色名称，不写单页构图，不写营销口号。

### `design_soul`

- 只描述整套视觉系统要建立的目标和判断标准。
- 不能写某一页的具体构图、具体物体、具体镜头。
- 必须让 design agent 知道“怎样判断这套风格是否成立”。

### `variation_strategy`

- 必须描述跨页如何变化，例如锚点尺度、材质稠密度、留白强度、节奏切换。
- 必须同时说明不允许变化的维度，例如字体角色、变量体系、主强调关系。
- 不能写成封面页怎样、目录页怎样的逐页脚本。

### `recommended_combos`

- 只写组合语法，例如“编号锚点 + 分隔线 + 角标”。
- 不写具体位置、具体尺寸、具体句子。

### `css_snippets`

- 只保留必须跨页重复的局部样式锚点。
- 不得包含整页布局、完整容器树、整段 HTML 结构。

---

## 4. 风格自由区

允许大胆变化的维度：

- 页面之间的视觉重量切换。
- 装饰出现频率和稀疏度。
- 强调色在不同页的使用位置。
- 同一视觉语法下的节奏差异。

不允许违约的维度：

- 跨页变量体系断裂。
- 标题、正文、注释的角色关系失真。
- 装饰抢走内容主阅读任务。
- 相邻页面变成同构复制。

---

## 5. 风格失败模式

### 风格漂移

- 触发条件：页面开始引入未被 `style.json` 授权的新视觉语法。
- 失败征兆：同一 deck 内出现两个不同的标题系统、材质体系或强调逻辑。
- 修复顺序：先收回变量与字体角色，再收回装饰组合，再调局部细节。

### 装饰替代内容

- 触发条件：装饰数量上升，但有效信息密度没有提升。
- 失败征兆：页面主阅读路径被线条、光晕、浮层占据，内容卡片承担不了论证任务。
- 修复顺序：先补 payload，再压缩装饰，再恢复锚点与支撑的比例。

### Deck 内风格断裂

- 触发条件：某些页脱离全局风格规则，单独追求戏剧化。
- 失败征兆：页面单看成立，放回整套 deck 时节奏断裂。
- 修复顺序：先对齐字体角色和变量，再对齐 signature move 的出场节奏。

### 过度模板化

- 触发条件：同一种装饰组合、同一种标题构图、同一种卡片骨架在多页机械复用。
- 失败征兆：相邻页面可互换，阅读者感知不到节奏推进。
- 修复顺序：优先改变锚点位置、卡片主次和留白关系，再调装饰密度。

## Playbook（执行细则）

# Style Phase 1 Playbook -- 风格合同的定义与输出

## 目标

基于用户需求、大纲结构和 runtime 风格规则，输出一份可被 planning/html 稳定消费的 `style.json` 全局风格合同，而不是任何单页的 HTML 代码。

---

## 阶段执行流程

### Phase 1: 提炼风格约束

1. 读取 `requirements-interview.txt`
2. 抽取高优先级信号：品牌色、品牌禁区、受众、正式度、语言、配图策略。绝不允许在不知情的情况下选错受众（如给技术向听众做可爱风）。
3. 读取大纲 `outline.txt`
4. 判断整套 deck 的节奏类型：稳态推进 / 波浪起伏 / 发布会式冲刺 / 培训式均匀展开，以决定你的变化策略基调。

### Phase 2: 创造核心风格调色盘（拒绝机械查表）

不要做生搬硬套的“填表员”！**用户的需求（风格、受众、场景）拥有绝对的第一优先级！**

**决策你的核心风格取向：**
1. **深度提炼用户采访需求**：优先读取 `requirements-interview.txt` 中归一化后的 `style`、`brand`、`audience`；若文件保留 canonical 字段，也要同时对齐 `visual_style`、`brand_constraints`、`core_audience` 的含义。
2. **高级动态调色盘**：摆脱老土的死板色彩。凭借你的高级审美，构建一整套逻辑自洽的 12 个关键 `css_variables`（主背景色、卡片梯度、4个 accent 强调色）。
3. **只在极度迷茫时，才参考预置文件**。
如果用户要求“神秘东方”、“末日废土”、或者“社区极客感”，**你可以为其量身定做色彩搭配。但是，任何原创配色必须秉持至高无上的 Web 视觉安全原则！底色必须低频或静谧，文字绝不允许丢失强烈的对比度可读性。严禁为了追求“特立独行”搭配出导致阅读瞎眼的乱色！**

哪怕用户需求罕见如“神秘东方”、“孟菲斯几何”、“末日废土”或者“技术社区感”，**也必须将它们的高级感封装在严苛的系统变量矩阵中！**。千万不要以“创意”为名配出违背美学法则的泥石流色组。用户的 `style` / `visual_style` 需求指南针拥有第一地位，而你的产出必须符合 Token 纪律与大厂视觉审美红线。

### Phase 3: 生成 style.json 合同

你必须输出一份严格遵守下列字段要求的 JSON 合同文件：

*   `style_id` / `style_name`
*   `mood_keywords`：**（强制：必须提供 3-5 个关键词的数组）**
*   `design_soul`：描述整套 deck 的设计目标，**绝对不可以**写成某一页的成品描述或构图指导。
*   `variation_strategy`：必须同时说明“哪些元素允许变”和“哪些元素锁死不动”。不能写成逐页执行指令。
*   `decoration_dna.signature_move`：必须有，且为非空字符串。
*   `decoration_dna.forbidden`：**（强制：必须提供 2-5 个元素的数组）**
*   `decoration_dna.recommended_combos`：**（强制：必须提供 2-4 个元素的数组）**
*   `font_family`

#### css_variables 键命名与数量规范（强制红线）

这 12 个变量是基石，必须定义并且键名**不能更改一个字母**：

```json
{
  "bg_primary": "#...",
  "bg_secondary": "#...",
  "card_bg_from": "#...",
  "card_bg_to": "#...",
  "card_border": "#...",
  "card_radius": "...px",
  "text_primary": "#...",
  "text_secondary": "#...",
  "accent_1": "#...",
  "accent_2": "#...",
  "accent_3": "#...",
  "accent_4": "#...",
  "css_snippets": {
    "example_class": "font-weight: 700;"
  }
}
```

- key 必须使用下划线（无 `--` 前缀），对应校验合同要求。
- 必须严格保留这 12 个基础变量名，禁止改名。如需自定义增加可以增加，但这 12 个不可少。
- `css_snippets` 必须是对象 (Object)，格式为 `"键名": "值"`，**绝对不能是数组 (Array)！** 确有必要时可用它固化跨页重复的局部样式结构（如阴影），但绝对不能包含能驱动整页骨架布局的 CSS。
